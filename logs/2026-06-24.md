# 2026-06-24

## Inventory scroll bar — finished (board: Systems & performance)

The scroll bar was half-scaffolded: `Inventory::page` existed, `filter_items` /
`set_slots` already paginated by it, and `ScrollButton` / `ScrollButtonBundle`
were defined — but the buttons were never spawned and nothing handled a click to
change the page. Wired up the missing half in `src/libs/inventory.rs`.

### Changes

- **Buttons spawned.** `InventoryBundle::spawn` now spawns a left (prev) and right
  (next) button as children of the inventory entity, just below the slot grid
  (`y = -inventory_size.y/2 - SCROLL_BUTTON_SIZE`). They're children, so they
  inherit the inventory's transform/visibility and sit in the same coordinate
  space as the slots.
- **No PNG assets — lyon triangles instead.** The old `ScrollButtonBundle`
  loaded `left_arrow.png` / `right_arrow.png`, which **do not exist** in `assets/`
  (would have failed at runtime). Rather than author binary arrow images,
  rebuilt the bundle to draw a filled triangle with `bevy_prototype_lyon`
  (`shapes::Polygon`, `ShapeBuilder::with(..).fill(..).stroke(..).build()`) —
  matching the project's established shape convention (button minigame, engage
  button). Triangle points left or right per the `left` flag.
- **Click handler.** New `handle_scroll_click` system (registered in `main.rs`
  Update chain, right after `handle_slot_click`). Mirrors `handle_slot_click`'s
  hit-test: on `mouse_state.just_released`, find a `ScrollButton` whose
  `RectangularArea` contains the click. Left decrements `page` (floored at 0);
  right increments only if a further page exists. Mutating `Inventory::page`
  trips `Changed<Inventory>`, which is exactly what `set_slots` keys on to
  repaint — so no extra redraw plumbing was needed.
- **`ScrollButton` carries its `inventory: Entity`** (like `Slot` carries
  `inventory`) so the handler knows which inventory to page. Made the component
  `pub` — it appears in the public system's query type, else
  `private_interfaces` warns.
- **Page-bound helper.** Added `count_filtered_items(store, filter) -> usize`
  (counts matches across all pages) to compute the last page
  (`total.saturating_sub(1) / per_page`) so "next" can't scroll past the end.
  Unit-tested.

### Verified in-game (scroll bar)

Robert confirmed paging works. To exercise it, temporarily seeded a chest in
`setup_board` with 20 distinct item types (4 forms × 5 materials) — the normal
starting board has nowhere near enough items to fill one page. Two gotchas that

`Item::draw`. The seed was removed after verification.

### life/land `Shape`-vs-`Sprite` query — corrected the type, but it's dormant

Fixed the migration-flagged bug: `life`/`land` `evolve_fixed_update` declared
`Query<&mut Shape, With<Cell>>`, but their cells render via `Sprite` (and
`CellBundle::turn_on/turn_off` already take `Query<&mut Sprite, With<Cell>>`).
The `Shape` query was lyon-era dead weight that matched nothing. Changed the type
to `Query<&mut Sprite, With<Cell>>`, renamed the param `fill_query` →
`cell_draw_query` (matching `cell_update`), and dropped the now-unused
`bevy_prototype_lyon` import from both files (it was their only lyon use).

IMPORTANT, found while fixing: this bug is currently **dormant** — the type fix
has no runtime effect today, because:
- `life::cell_update`, `life::evolve_fixed_update`, `land::cell_update`, and
  `land::evolve_fixed_update` are **not registered** in `main.rs` at all. They
  never run.
- `life` isn't in the `unlocks` table (minigame.rs), so it's never even spawned.
  `land` is unlockable (ocean L1) but its systems still don't run.
- `life::evolve_fixed_update` is a `return; // TODO` stub; `ingest_fixed_update`
  is empty. `land::evolve` runs the sim on the data model but has no rendering
  step (the `cell_draw_query` param is unused scaffolding).

So making evolution actually visible is feature work (board: "Finish the
incomplete minigame implementations"), not a cleanup bug. The type fix just makes
the scaffolding correct for whoever wires it up.

Other concern at this level: `life.rs` and `land.rs` each start with
`#![allow(warnings)]`, which silences ALL warnings — that is exactly what hid the
`Shape`/`Sprite` mismatch, the unused query params, and the dead lyon imports.
No other file does this (main.rs has it commented out). Recommend removing both
and cleaning the cascade as a dedicated step.

### Fixed `RectangularArea::nearest_edge` (board: "Area nearest-point")

The old implementation snapped both axes to extremes (always a corner) and
inverted y (a point below center returned the top edge). Two real call sites:
`mouse.rs` drag-clamping (snapped dragged entities to a corner instead of
sliding along the edge) and `minigame.rs` `clear_clutter` (shoved overlapping
items to a corner).

Rewrote it as a proper nearest-perimeter-point: clamp the point into the rect
(an outside point lands on the boundary), and if the point was inside, project
to whichever of the four edges is closest. Dead-center is equidistant, so the
tie-break deterministically picks the left edge ("arbitrary point on the edge"
per the doc). `clamp` now slides outside points onto the edge instead of a
corner.

Test-first: replaced the `..._known_limitation` test (which pinned the buggy
corner+y-flip) with four tests — outside-clamps-to-boundary (incl. the diagonal
corner case), inside-projects-to-closest-edge (incl. an explicit y-inversion
regression guard), center-picks-deterministic-edge, and clamp-outside-slides-to-
edge. 33 tests pass, clippy clean. Removed the board task. Not yet verified
in-game (drag a follow-mouse entity past its bounds to see the edge-slide).
