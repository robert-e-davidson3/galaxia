## Made `nix develop --command` usable (flake)

The `flake.nix` shellHook ended with an unconditional `exec zsh`, which hijacked `nix develop --command ...`: the `exec` replaced the process before the passed command ran, so cargo never executed (a probe command printed nothing). Guarded it with `if [[ $- == *i* ]]` so the interactive drop-into-zsh only fires for interactive shells; `--command` (and CI/tooling) now runs the command. The env exports (`LD_LIBRARY_PATH` / `PKG_CONFIG_PATH`) stay before the guard, so they still apply in `--command` mode too. Verified: `cargo --version` and `pkg-config --exists alsa` both resolve through `--command`.

## Clippy triage (89 → 39)

Clippy reported 89 warnings. Robert ran `cargo clippy --fix` (~41 mechanical fixes — clone-on-`Copy`, manual `Option::map` → `.map`, `match` → `if let` / `matches!`, needless borrow/return, same-type casts, etc.) and committed. Then silenced `too_many_arguments` (9) crate-wide via a `[lints.clippy]` table in `Cargo.toml` — it fires constantly on Bevy systems, where a high parameter count (many `Query`/`Res`) is normal and idiomatic; allowing it centrally beats scattering `#[allow]` or contorting systems. Net **89 → 39**.

The remaining 39 are (a) the dead-code warnings we deliberately keep — unfinished feature surface (unused mana/energy variants, the half-built scroll-bar scaffolding), left for the same reason as the unused deps: "unused" here means "not wired up yet," not garbage — and (b) a handful of non-auto-fixable style lints, notably `type_complexity` (the `Arc<Mutex<HashMap<…>>>` / big Bevy query tuples), which need a human-chosen `type` alias.

## Graphics: removed wayland (this is an X11 system)

`cargo run` initially panicked: every mesa Vulkan ICD failed to load with `undefined symbol: wl_display_dispatch_queue_timeout` (a `libwayland-client` symbol), so Vulkan found no drivers and wgpu's GLES/EGL fallback panicked on an `unwrap()`. Root cause: the flake put its own (older, pinned) `wayland` first on `LD_LIBRARY_PATH`, **shadowing** the system's newer `libwayland-client` that the system mesa (25.2.6, NixOS 25.11) needs. Robert no longer uses wayland (X11 now), so removed `wayland` from the flake's `buildInputs` — mesa then resolves its own matching wayland via its runpath, and the X11 app never needs libwayland directly. Kept `libxkbcommon` (X11 still uses it for keyboard). After re-entering the shell, the game runs: `radv` / Vulkan adapter on the Radeon 780M, X11 window, gameplay confirmed (`Player is now sticky`). This also confirms the `opt-level = 1` dev profile is playable.

## Observed during the run: a B0003 double-despawn (minor, untracked)

Bevy logged `error[B0003]: Could not despawn entity … because it doesn't exist` — a despawn of an already-gone entity. Non-fatal, but a real minor bug. Prime suspect: `combine_loose_items`, which despawns both colliding items, so two collisions touching the same item in a single frame would despawn it twice. Not yet on the board.

Also benign: two `VALIDATION [VUID-StandaloneSpirv-…]` errors from Bevy's own built-in compute shaders (the mip-downsampling `previous_mip_depth` workgroup array), flagged by the Vulkan validation layer; radv runs them fine. A known Bevy-0.14-era thing — likely clears on the Bevy upgrade already on the board.
