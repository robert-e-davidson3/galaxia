## Made `nix develop --command` usable (flake)

The `flake.nix` shellHook ended with an unconditional `exec zsh`, which hijacked `nix develop --command ...`: the `exec` replaced the process before the passed command ran, so cargo never executed (a probe command printed nothing). Guarded it with `if [[ $- == *i* ]]` so the interactive drop-into-zsh only fires for interactive shells; `--command` (and CI/tooling) now runs the command. The env exports (`LD_LIBRARY_PATH` / `PKG_CONFIG_PATH`) stay before the guard, so they still apply in `--command` mode too. Verified: `cargo --version` and `pkg-config --exists alsa` both resolve through `--command`.

## Graphics: removed wayland (this is an X11 system)

`cargo run` initially panicked: every mesa Vulkan ICD failed to load with `undefined symbol: wl_display_dispatch_queue_timeout` (a `libwayland-client` symbol), so Vulkan found no drivers and wgpu's GLES/EGL fallback panicked on an `unwrap()`. Root cause: the flake put its own (older, pinned) `wayland` first on `LD_LIBRARY_PATH`, **shadowing** the system's newer `libwayland-client` that the system mesa (25.2.6, NixOS 25.11) needs. Robert no longer uses wayland (X11 now), so removed `wayland` from the flake's `buildInputs` ‚Äî mesa then resolves its own matching wayland via its runpath, and the X11 app never needs libwayland directly. Kept `libxkbcommon` (X11 still uses it for keyboard). After re-entering the shell, the game runs: `radv` / Vulkan adapter on the Radeon 780M, X11 window, gameplay confirmed (`Player is now sticky`). This also confirms the `opt-level = 1` dev profile is playable.

## Crash fix: camera unwrap on a stale engaged-minigame entity

`cargo run` crashed in `update_camera` (`camera.rs`): `called Result::unwrap() on an Err value: NoSuchEntity`. Cause: `Engaged.game` holds the focused minigame's `Entity`, but when that minigame **levels up**, `levelup` (`minigame.rs`) despawns it and respawns a new entity without updating `Engaged` ‚Äî so the camera unwrapped a dangling entity the next frame. (The repeated `B0003` despawn warnings just before the panic are a related, non-fatal stale-handle bug ‚Äî now boarded.)

Two fixes:
- **`camera.rs`** ‚Äî `update_camera` guards the engaged lookup (`if let Ok(...)` instead of `.unwrap()`); a missing engaged minigame falls through to following the player rather than panicking. Defensive code that stays regardless of the below.
- **`minigame.rs` `levelup`** ‚Äî clears `Engaged.game` when the engaged minigame levels up, so nothing dangles.

Compiles clean, clippy still 0, and Robert confirmed it no longer crashes.

**The levelup behavior is wrong by design, though** (Robert): the camera should **keep** focus through a levelup, not drop it ‚Äî early levels happen fast and losing focus is jarring. So the `Engaged`-clear is a stopgap. Next (board ‚Üí Now): remap `Engaged` to the respawned minigame and reconcile the new engage button's toggle/fill state so focus persists across the despawn/respawn. The `camera.rs` guard stays either way.

## Camera focus tracks the minigame by id (survives levelup)

Implemented the real focus-retention, replacing the stopgap above, with the model Robert chose: focus is "which minigame," not "which entity instance."

- **`Engaged.game` is now `Option<&'static str>`** (the minigame id) instead of `Option<Entity>`; `Minigame::id()` returns `&'static str` (it already returned the static `ID` consts). `Engaged` stays `Copy`.
- **The camera resolves the id to the live entity** via `MinigamesResource::entity(id)` (with the stale-entity guard), so the same id keeps focusing the minigame across its despawn/respawn.
- **`levelup` re-registers the respawned minigame** (`set_entity(id, new_entity)`) and no longer clears focus ‚Äî since the id is stable, focus persists automatically.
- **The engage button is now derived state.** Dropped its `Toggleable` (a rival source of truth); `engage_button_update` flips `Engaged` by id on click, and a new `update_engage_button_appearance` system sets each button's fill from `Engaged` every frame, so a respawned button immediately shows the right engaged/un-engaged look.
- Tidied the `MinigamesResource` id-lookup methods (`entity`/`set_entity`/`is_unlocked`/`prerequisites`/`to_unlock`/`needs_to_unlock`/`unlocked_by`/`level`) to take `&str`, consistent with `insert`, dropping a couple of needless `.into()`s.

### Two bugs caught in playtesting ‚Äî both latent, surfaced by making `MinigamesResource` load-bearing

1. **Dropping focus on levelup was wrong by design** ‚Äî superseded by the id model above (the camera now stays on the minigame for free).
2. **Startup registration never happened**, so focus worked only *after* a levelup or unlock. Root cause: `setup_board` (which `set_entity`s the three startup minigames) and `setup_minigame_unlocks` (which `insert`s the keys `set_entity` needs) were both unchained `Startup` systems sharing `ResMut<MinigamesResource>`, so Bevy ran them in an arbitrary order ‚Äî and `setup_board` ran first, so every startup `set_entity` silently no-op'd on an unknown key (`entity(id)` stayed `None` until a levelup re-registered it). Fixed by `.chain()`-ing the `Startup` tuple (unlocks before board). Bonus: this also fixes the latent `is_unlocked`-at-startup staleness (the three base minigames read as "not spawned" in the registry until their first levelup). Same theme as the dependency / `MinigamesResource` staleness throughout this session: a value that's harmlessly stale becomes a real bug the moment something starts trusting it.

Confirmed working in-game: focus grabs immediately on the fresh startup minigames and persists through levelups; click-off leaves focus off.

## Root-caused and fixed the `B0003` double-despawn on levelup

The repeated `error[B0003]: Could not despawn entity ‚Ä¶ because it doesn't exist` on minigame levelup was **not** a same-frame double-despawn (the earlier `combine_loose_items` suspicion was a red herring ‚Äî that system already guards itself with an `eliminated` set, and loose items are top-level entities, not minigame children). The real cause is a **stale `Children` reference left by plain `despawn()` on a child entity.**

Traced through the Bevy 0.14.2 source to confirm the mechanism:
- `despawn_recursive()` ‚Üí `despawn_with_children_recursive` (`bevy_hierarchy/src/hierarchy.rs:24`) **removes the entity from its parent's `Children` first** (lines 26‚Äì30), then recurses.
- Plain `commands.entity(e).despawn()` does **not** touch the parent's `Children` ‚Äî it just removes the entity, leaving a dangling child handle.
- The recursive walk (`despawn_with_children_recursive_inner`, line 37) iterates the parent's `Children` and calls `World::despawn` on each, which **warns `B0003` at `bevy_ecs/src/world/mod.rs:1108`** when the target no longer exists.

So the chain is: a minigame's children (ball_breaker **blocks**/**balls**, tree **fruits**, and the engage button's **hover text**) are destroyed mid-play with plain `despawn()`, each leaving a stale entry in the parent's `Children`. When the minigame later levels up, `levelup` (`minigame.rs:398`) calls `despawn_recursive()` on it, the recursion walks the stale `Children`, and every already-gone child emits one `B0003` warn ‚Äî hence the *repeated* errors, one per child destroyed that level. In ball_breaker this is guaranteed: breaking the **last** block (`ball_breaker.rs:582`) both despawns it and inserts `LevelingUp` in the same flow, so the levelup that follows always walks a `Children` full of broken blocks.

Fix ‚Äî destroy minigame-descendant children with `despawn_recursive()` (which detaches from the parent) instead of `despawn()`. Four sites changed:
- `ball_breaker.rs` ‚Äî block despawn and ball despawn.
- `tree.rs` ‚Äî picked-fruit despawn.
- `mouse.rs` `update_hover_text` ‚Äî the un-hover text-child despawn (the hover text is a child of `MinigameEngageButton`, itself a child of the minigame).

Left as plain `despawn()` deliberately: `mouse.rs` `ClickIndicator` (top-level, no parent) and `item.rs` `combine_loose_items` (loose items are top-level, and that system is already self-guarded).
