### Why this shape

From Sorcida's `CLAUDE.md`: logs are append-only history (nuance over time), references are the current settled truth (promote a fact out of a log once it stops moving), skills are procedures written so an AI can run them cold, and the board is forward-only (finished work goes to the log, not a "Done" column). The split keeps `CLAUDE.md` small and the durable facts findable.

## Build profile, package rename, dependency audit, and first geometry tests

A round of repo-health changes that came out of a "what do you actually see to do here" survey (separate from the migrated board). All of it was authored with no compiler available, so it's queued for verification (see the status note at the end).

### Dependency audit — surveyed, deliberately NOT pruned

Grepping `src/` for each declared crate, these appear **zero** times: `array2d`, `ndarray`, `perlin_noise`, `libnoise`, `once_cell`, `imageproc`, a *direct* `rapier2d` (only `bevy_rapier2d` is used), and `serde`/`serde_json` (no `Serialize`/`Deserialize` derives anywhere). That looked like ~half the dependency list was dead weight — **but Robert flagged that several were used for generating the resource images, and some were for other game directions he didn't pursue.** So they stay. Recorded here so a future "remove unused deps" pass doesn't blindly delete them — confirm intent first; the grep-zero result is about *current* `src/` references, not whether they were ever load-bearing.

### Dev build profile (committed, needs play-test)

`Cargo.toml` had `opt-level = 3` on the dev profile, so our own crate was fully optimized on every compile — slow incremental builds. Switched to the standard Bevy split: `[profile.dev] opt-level = 1` for our code, `[profile.dev.package."*"] opt-level = 3` for dependencies, so Bevy/physics/rendering stay fast (the game should still be playable) while our code compiles quickly. Release unchanged. **Unverified:** needs a real `cargo run` to confirm our own systems aren't sluggish at `opt-level = 1`; bump to `2` if they are. Robert is marking the commit as needing testing.

### Package rename

`Cargo.toml` `name` was `test_project` (a `cargo init` leftover); renamed to `galaxia`. The only other occurrence was `Cargo.lock` (regenerates on the next build); no `.rs`/`.nix`/script references, since internal code uses `crate::`.

### First pure-logic tests: `src/libs/area.rs`

Before this, only `item.rs` had any tests. Added a `#[cfg(test)] mod tests` to `area.rs` (14 tests) covering the geometry that's pure and window-free: `RectangularArea` and `CircularArea` (`is_within`, `overlaps`, `nearest_edge`, `clamp`, `grow`, edges/dimensions), the `Area` enum's mixed-type overlap path, and the `From` conversions. Matched the existing test style (`use super::*;`). Expected values were hand-traced since there was no compiler. One test — `rect_nearest_edge_returns_a_corner_known_limitation` — intentionally pins the *current* buggy behavior (rect `nearest_edge` snaps to a corner and inverts y); it's commented to be updated when the `area.rs:218` TODO / the "Area nearest-point" board task is fixed, so the fix will trip it as a reminder rather than a surprise. These cover `area.rs`; the higher-value `inventory.rs::filter_items` test (which would cover the earlier unverified rewrite) is harder — `HashMap` iteration order is nondeterministic, so a pagination test must assert counts, not item order — and is left as follow-up.
