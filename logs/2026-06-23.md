### 0.14 → 0.15 changes applied (73 compile errors, all mechanical patterns)

- **`Time` method renames:** `elapsed_seconds()` → `elapsed_secs()`, `delta_seconds()` →
  `delta_secs()` (foundry, rune, tree, mouse, camera).
- **`SpriteBundle.texture` removed:** the image moved into `Sprite.image`. `SpriteBundle`
  still implements `Bundle` in 0.15 (deprecated), so the minimal fix is to keep the bundle
  and move `texture` into the inner `Sprite { image, .. }`. Done in item, ball_breaker,
  tree, inventory. The full split to a `Sprite` component is deferred to step 2, when
  `SpriteBundle` is *removed*.
- **`Handle<Image>` is no longer a `Component`:** in 0.14 a `SpriteBundle` gave the entity a
  separate `Handle<Image>` texture component, which `land`/`life` queried as
  `(&mut Handle<Image>, &mut Sprite)` to repaint cells. In 0.15 the image is `Sprite.image`,
  so those queries became `&mut Sprite` and the writes became `sprite.image = h`. Same idea
  in inventory: `present_sprite` now takes the handle and sets `Sprite.image` instead of
  inserting a bare `Handle` component.
- **Text rework (hard break — `Text2dBundle` lost `Bundle`):** replaced
  `Text2dBundle { text: Text::from_section(s, TextStyle{..}), .. }` with the component tuple
  `(Text2d::new(s), TextFont{font_size,..}, TextColor(c), [TextLayout::new_with_justify(j)],
  Transform)`. Updating live text is now `*Text2d` / `text.0 = ..` (button click counter),
  and the query changed `&mut Text` → `&mut Text2d`. Done in minigame, button, mouse.
- **lyon `ShapeBundle.spatial` removed:** the `spatial: SpatialBundle{ transform, .. }` field
  is gone; `ShapeBundle` now has `transform`/`visibility` (plus `mesh`/`material`) directly,
  so `spatial: SpatialBundle{ transform: T, ..}` → `transform: T`. `shapes::Rectangle` also
  gained `radii: Option<BorderRadii>` — added `..default()` where the literal didn't have it.
  Done in minigame, button, rune, mouse.
- **Rapier multi-world refactor (bevy_rapier 0.29):** `RapierConfiguration` is now a
  *component* on the auto-spawned default context entity (seeded in `PreStartup`), and
  `TimestepMode` is a separate `Resource`. The old `insert_resource(RapierConfiguration{..})`
  is gone; replaced with a `Startup` system `setup_physics` that zeroes gravity on the
  context (the only field we deviate from defaults — the default `TimestepMode` already
  matches our old Variable config). `RapierContext` is also no longer a resource: `grab_items`
  switched `Res<RapierContext>` → the `ReadRapierContext` system param and calls
  `read_rapier_context.single().contact_pair(..)`.
- **Misc:** `commands.add` → `commands.queue`; `World::get_entity_mut` now returns `Result`
  (`Some` → `Ok`); `Camera::viewport_to_world` now returns `Result` (added `.ok()` in the
  `.and_then`); `ImageSampler` import moved `bevy::render::texture` → `bevy::image`.

## Bevy upgrade — Step 2 of 4: 0.15 → 0.16

Robert opted to continue straight on without an in-game check of 0.15. This is the biggest
step: 0.16 *removes* the bundles that 0.15 only deprecated, and bevy_prototype_lyon 0.14
rewrote its whole shape API. ~190 compile errors, fixed by fanning out one subagent per
file-group against a shared mechanical spec, with the nuanced files done by hand.

### 0.15 → 0.16 changes applied

- **Deprecated bundles removed → components:**
  - `SpriteBundle` → `Sprite` + `Transform`. The user Bundle structs that nested
    `sprite: SpriteBundle` (ItemBundle, BlockBundle, BallBundle, PaddleBundle, CellBundle×2,
    UnpickedFruitBundle, SlotBundle, ScrollButtonBundle, SearchBoxBundle) each became two
    fields `sprite: Sprite, transform: Transform`. Inline `SpriteBundle{..}` spawns →
    `(Sprite{..}, Transform)`.
  - `SpatialBundle` → `(Transform, Visibility::default())` (or two struct fields).
  - `TransformBundle` → plain `Transform`.
  - `Camera2dBundle` → `Camera2d`.
- **lyon 0.13 → 0.14 (big rework):** `ShapeBundle`, `GeometryBuilder`, and the `Fill`/`Stroke`
  *components* are all gone. There is now a single `Shape` component built with a builder:
  `ShapeBuilder::with(&geo).fill(Fill::color(c)).stroke(Stroke::new(c,w)).build()`. `Fill`/
  `Stroke` are plain structs living in `Shape.fill`/`Shape.stroke` (Options). So:
  `Query<&mut Fill>` → `Query<&mut Shape>` and `fill.color = c` → `shape.fill = Some(Fill::color(c))`
  (rune pixels, land/life cell fill, the engage-button alpha pulse, the mouse ClickIndicator).
  `Shape` `#[require(...)]`s Transform/Visibility/Mesh2d/material, so a non-default transform
  is spawned as its own tuple element alongside the built `Shape`.
- **Hierarchy rename:** `Parent` → `ChildOf` (`.get()` → `.parent()`); the `with_children`
  closure param type `ChildBuilder` → `ChildSpawnerCommands`; `spawner.parent_entity()` →
  `spawner.target_entity()`.
- **Camera projection:** `OrthographicProjection` is no longer a standalone component; the
  camera carries a `Projection` enum. `update_camera` queries `&mut Projection` and pulls the
  ortho out with `let Projection::Orthographic(p) = projection.as_mut() else { return };`.
- **`Query::single`/`single_mut` now return `Result`** (old panicking `single` gone, `get_single`
  deprecated). Bare `.single()` call sites that returned the item now use `let Ok(x) = .. else
  { return ..; }` (mouse `get_mouse_position`, camera, item `grab_items`).
- **`Component` mutability bound:** `EntityWorldMut::get_mut::<T>()` now needs
  `T: Component<Mutability = Mutable>`; added that bound to `mark_component_changed` (misc.rs).
- **rapier 0.29 → 0.31:** `ReadRapierContext::single()` now returns `Result` — `grab_items`
  unwraps with `let Ok(rapier_context) = .. else { return; }`.
- **`despawn_recursive` → `despawn`** (plain despawn is recursive in 0.16; the B0003 fix sites
  in minigame/ball_breaker/tree/mouse). **`EventWriter::send` → `write`** (main exit_system).

### Status

`cargo check` clean (0 errors, 0 deprecation warnings remaining), `cargo test` 25/25 green.
Not yet verified in the running game. Subagent flag worth noting: land/life
`evolve_fixed_update` now hold a `Query<&mut Shape, With<Cell>>` that matches nothing at

## Bevy upgrade — Step 3 of 4: 0.16 → 0.17

A small step in code terms; the toolchain + a new system dep were the real work.

- **Toolchain:** `nix flake update rust-overlay` re-locked the rust-overlay input
  (2025-06-25 → 2026-06-23), moving `rust-bin.stable.latest.default` from **1.87.0 → 1.96.0**
  — clears the 1.88 (0.17) and 1.89 (0.18) MSRVs in one bump. Gotcha: the interactive dev
  shell caches the old toolchain, so it still reports 1.87 until re-entered. Build/test for
  the rest of the migration must run via `nix develop --command cargo …` to pick up 1.96.
- **New system dep (flake.nix):** Bevy 0.17's winit links `wayland-client` by default, so the
  first build failed in `wayland-sys`'s build script ("`wayland-client.pc` not found"). Added
  `wayland` to the flake `buildInputs` (covers both `PKG_CONFIG_PATH` and the runtime
  `LD_LIBRARY_PATH`, which the shellHook derives from `buildInputs`).
- **Code (only two renames):**
  - `JustifyText` → `Justify` (the text justify enum was renamed). `minigame.rs`.
  - `RenderAssetUsages` moved out of `bevy::render::render_asset` (now private there) to
    `bevy::asset::RenderAssetUsages`. `images.rs` (both import sites).

`nix develop -c cargo check` clean (0 errors), `cargo test` 25/25 green. Note: the editor's
rust-analyzer throws false errors here (e.g. "expected &mut Assets, found &mut ResMut",
"proc-macro not yet built") because it still runs under the cached 1.87 shell and can't build
0.17's proc-macros; the real compiler under 1.96 disagrees. Not yet verified in the game.

## Bevy upgrade — Step 4 of 4: 0.17 → 0.18.1  (migration complete)

No toolchain or flake change (Rust 1.96 already clears 0.18's 1.89 MSRV). Two breaking changes:

- **Event → Message rename (the big 0.18 change):** buffered events are now "messages."
  `EventReader<E>` → `MessageReader<M>`, `EventWriter<E>` → `MessageWriter<M>`. Semantics are
  identical — `read()` still yields `&M`, so the existing `*entity` / `.contains(entity)` code
  in the collision handlers type-checks unchanged once the param type resolves. Sites:
  `CollisionEvent` readers (item `combine_loose_items` + `grab_items`, minigame, ball_breaker),
  `MouseWheel` reader (camera), `AppExit` writer (main). `MouseWheel`/`AppExit`/rapier's
  `CollisionEvent` are all `Message` types in their 0.18 builds.
- **rapier 0.34 renamed `Velocity` fields:** `linvel` → `linear`, `angvel` → `angular`
  (item.rs combine, constant_velocity.rs). Constructors `Velocity::linear(..)`/`angular(..)`
  are unchanged.

`nix develop -c cargo check` clean (0 errors), `cargo test` 25/25 green under Rust 1.96.

## Runtime GPU crash after the migration — wayland version skew (fixed)

First in-game run (`cargo run` under X11/i3) built fine but panicked at GPU init: every system
mesa Vulkan ICD (`libvulkan_radeon.so` etc., from the host's `mesa-25.2.6`) failed to load with
`undefined symbol: wl_display_dispatch_queue_timeout` → "Found no drivers!" → "Unable to find a
GPU!".

**Cause:** the `wayland` I added to `flake.nix` buildInputs for the 0.17 build went onto
`LD_LIBRARY_PATH` ahead of the system's libwayland. But the flake's nixpkgs was still pinned to a
mid-2025 `nixos-unstable` whose wayland was **1.23.1**, which lacks `wl_display_dispatch_queue_timeout`
(a newer symbol the host's mesa 25.2.6 needs). Our older libwayland shadowed the system's, so the
mesa drivers couldn't resolve the symbol. (winit 0.30 hard-links libwayland — `wayland` is a default
Bevy feature, bundled with `x11` inside the `2d`/`3d` meta-features via `default_platform`, so it
can't be dropped without `default-features = false` surgery. We didn't go that route.)

**Fix (Robert's call):** point the flake's nixpkgs at the system's channel instead of unstable —
`nixpkgs.url = github:NixOS/nixpkgs/nixos-25.11` + `nix flake update nixpkgs`. The flake's wayland
is now **1.24.0** (has the symbol; matches the host), vulkan-loader 1.4.328. Rust stays 1.96
(rust-overlay is a separate input). `cargo build` links cleanly under the new libs.

Net flake state after the migration: nixpkgs → `nixos-25.11`, rust-overlay re-locked (Rust 1.96),
`wayland` added to buildInputs.

## Mass of B0004 warnings on run (fixed)

The game ran but spammed `warning[B0004]` (one per minigame-grid child) every startup. B0004 in
0.18 is registered on **`GlobalTransform`** (`bevy_transform .../global_transform.rs`: `on_insert =
validate_parent_has_component::<GlobalTransform>`): a child gets a `GlobalTransform` whose parent
has none.

Cause was a spawn-ordering latent bug, newly surfaced by 0.18's stricter check (not a 0.14→0.18
regression per se). Two sites used `commands.spawn_empty().with_children(|p| …spawn grid…)
.insert(SomeBundle)` where `SomeBundle` carries the parent's `Transform`. Because the bundle is
inserted *after* `with_children`, every child's `GlobalTransform` on-insert hook ran while the
parent still had no `Transform` → one warning per child (hundreds, for a cell/pixel grid).

Fixed both by giving the parent its spatial components *before* spawning children:
- `Minigame::spawn` (minigame.rs ~237): `spawn_empty()` → `spawn((transform, Visibility::default()))`
  (`Transform` is `Copy`, so it's still moved into the `MinigameBundle` insert afterward, which
  harmlessly re-sets the same values).
- `InventoryBundle::spawn` (inventory.rs ~48): same shape — spawn `(Transform, Visibility)` from
  `position` up front, then children, then the full bundle. (This one fired per-slot on chest/
  battery engage, not at startup.)

`cargo check` clean. Confirmed by Robert: B0004 spam gone, GPU now initializes (RADV / Vulkan /
Mesa 25.2.6 — the wayland fix held), window shows the game and it runs fine.

## Vulkan validation-layer ERROR spam (silenced)

With the game rendering, the console still spammed `ERROR wgpu_hal::vulkan::instance: VALIDATION
[VUID-StandaloneSpirv-MemorySemantics-10871]` — Bevy 0.18's GPU-preprocessing compute shaders do
`OpAtomicIAdd`/`OpAtomicStore` with relaxed memory order, which the Vulkan validation layer
(stricter since the nixos-25.11 pin) reports as a SPIR-V spec violation. It's non-fatal — RADV
runs the shaders fine — and it's upstream Bevy/wgpu, not galaxia code.

Per Robert's rule ("ERROR in output always warrants fixing — if it's noise, silence it"), disabled
the validation layer in the flake `shellHook`:
`export VK_LOADER_LAYERS_DISABLE=VK_LAYER_KHRONOS_validation`. wgpu only requests the layer in
debug builds; the loader now skips it, so no validation reporting. We write no raw Vulkan (Bevy
does), so dropping validation costs us nothing. Takes effect on a fresh `nix develop` (shellHook
only runs on shell entry).
