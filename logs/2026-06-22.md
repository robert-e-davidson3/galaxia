# 2026-06-22

## Removed the `Arc<Mutex<HashMap<ItemType, f32>>>` inventory storage

The board's tech-debt item asked to evaluate *why* the chest/battery item store was an `Arc<Mutex<HashMap<ItemType, f32>>>` and refactor if it was just working around the borrow checker. Evaluated, found it was, and removed it.

### What it was doing

The same `Arc` was held in two places: the minigame struct (`ChestMinigame`/`BatteryMinigame`) and the `Inventory` component on a *separate* inventory entity (`Inventory::new` cloned the `Arc`). Both sides mutated the shared map through the lock — the minigame side via `ingest_item` → `add_item`, the UI side via `set_slots`/`filter_items` (read) and `handle_slot_click` → `remove_item`.

Two real reasons it existed:

1. **Persistence across levelup.** On levelup the minigame entity and its child inventory entity are despawned and respawned, but the *struct value* survives (`levelup()` does `..self.clone()`). So the store had to live on the persistent struct, not the throwaway inventory entity. This is load-bearing.
2. **Mutation hidden from Bevy's scheduler.** Both sides mutated through *shared* (`&`) refs via the lock, so Bevy's access tracker couldn't see the writes — which is exactly why the `Mutex` was needed for correctness *given that design*. Classic case of opting out of ECS conflict detection and then having to hand-manage locking.

### The refactor (idiomatic ECS access)

Made the **minigame struct the sole owner** of a plain `HashMap<ItemType, f32>` (it's already the thing that persists across levelup), and had the inventory UI reach it through the owning entity:

- `ChestMinigame`/`BatteryMinigame`: `items: HashMap<ItemType, f32>` (dropped `Arc<Mutex>`); removed the `std::sync` imports.
- `Inventory`: dropped its `items` field — it's now pure view state (layout + filter + page). It already stored `owner: Entity`.
- New `Minigame::items()` / `items_mut()` accessors (`minigame.rs`) return `Option<&HashMap>` / `Option<&mut HashMap>`, matching the inventory-bearing variants (chest, battery) and `None` otherwise.
- Free fns `add_item`/`remove_item` now take `&mut HashMap`; `total_stored`/`filter_items` take `&HashMap` — all the lock-poisoning `.unwrap()`s are gone.
- `set_slots` gained `Query<&Minigame>` and reads items via `inventory.owner` → `minigame.items()`.
- `handle_slot_click` switched its minigame query to `&mut Minigame` and removes items via `items_mut()` (capturing `minigame.area()` and the `GlobalTransform` before taking the mutable borrow, then spawning the ejected item after).
- `InventoryBundle::spawn` takes the store by `&HashMap` to populate initial slots; `Inventory::new` dropped its items param.
- The `mark_component_changed::<Inventory>` pokes in chest/battery `ingest_item` stay — change detection still keys on `Inventory`, which is what `set_slots` watches.

The win: one owner, no lock, no poisoning `.unwrap()`s, and **Bevy now sees the access** — `ingest_item` (`&mut Minigame`), `handle_slot_click` (`&mut Minigame`) and `set_slots` (`&Minigame`) are scheduled with real conflict detection instead of hidden shared mutation. Levelup persistence still works for free via the struct clone (`HashMap` clones its contents).

### Tests first (then the refactor)

Per Robert's request, expanded coverage *before* refactoring so the assertions could be held constant across the change as a behavior-preservation proof. Added a `#[cfg(test)] mod tests` to `inventory.rs` (7 tests: `add_item` accumulation, `remove_item` partial/full/absent with key-drop-on-empty, `total_stored`, and `filter_items` empty-filter / pagination-coverage / uid-substring) and one to `chest.rs` (`levelup_preserves_stored_items`, covering reason 1 above). They passed against the old `Arc<Mutex>` API, then — with only the test *setup* lines changed (`store()` returns a plain `HashMap`; mutating calls take `&mut`), never the `assert_*` lines — passed identically against the new `HashMap` API. Suite went 17 → 25.

`cargo build` / `clippy --all-targets` (0 warnings) / `fmt --check` / `test` (25/25) all green. **Confirmed in-game** (Robert): items show in a chest/battery inventory on ingest, eject on slot-click, and survive a chest/battery levelup. Refactor complete and verified end-to-end.

## Bevy upgrade — recon only, then deferred

Started the Bevy stack upgrade; did the version/compatibility recon, then deferred the actual migration (too command-heavy for one sitting). Findings recorded on the board (`tasks/board.md` → Tech debt) so it's pick-up-ready; key facts:

- **Ecosystem ceiling is Bevy 0.18.** Latest stable Bevy is 0.19.0, but `bevy_rapier2d` (0.34→0.18.1), `bevy_prototype_lyon` (0.16→0.18.0), `bevy_ecs_tilemap` (0.21→0.18), and `bevy_framepace` (0.18.1→0.18.0) all top out at 0.18 — none support 0.19 yet (checked the crates.io sparse index 2026-06-22). So **target Bevy 0.18.1**, not 0.19.
- **Four-version jump** (0.14→0.18) — recommend stepping one version at a time, green at each, per the official migration guides.
- **MSRV bump:** Bevy 0.17 needs Rust 1.88, 0.18 needs 1.89; the flake pins 1.87.0, so the flake's Rust pin must move to ≥1.89.
- Watch-outs: standalone `rapier2d` must track `bevy_rapier2d` 0.34; `despawn_recursive` deprecated in 0.16 (affects today's B0003 fix); `flake.nix` system deps may shift.

Process note: the recon was run as many separate `curl` calls to the sparse index, each needing manual approval — too noisy. For a task like this, batch the read-only probing into as few calls as possible (or skip what isn't strictly needed). The version table above is enough to start; no further recon required.
